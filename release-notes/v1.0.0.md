# Release Notes - v1.0.0

**Release Date:** April 3, 2026

## 🎉 New Features

### Smarter Project Resolution

The issuer can now resolve projects more flexibly during authentication flows:

- **Project loading from request context** - When `client_id` is not explicitly provided, the server can resolve the project from the request host via the configured auth endpoint URL.
- **Improved multi-origin handling** - Follow-up fixes from `v0.3.1` stabilize flows where a project is accessed from multiple origins or hostnames.
- **Better passkey and callback origin matching** - Authentication flows are more tolerant of real deployment topologies and less dependent on a single hard-coded origin.

### Authentication Flow Improvements

- **Refined provider parsing** - OAuth and OIDC provider responses are parsed more consistently, reducing provider-specific edge cases during login.
- **Automatic user merge by email** - When a provider exposes an email and the relevant scope is available, existing users can now be reconciled by email instead of creating duplicate accounts.
- **Richer stored provider data** - User records now preserve provider-specific data more cleanly, while still keeping normalized top-level fields such as `email`, `name`, and `provider`.

### Email Template and Delivery Upgrades

- **Project-linked email templates** - Providers can now select templates by ID, allowing more explicit per-project email template configuration.
- **Sandboxed template functions** - Email template props now support `function::` helpers executed inside an isolated QuickJS sandbox.
- **TypeScript support in template helpers** - Template helper functions are transpiled before execution, making TypeScript-based template logic possible.
- **Structured custom delivery hooks** - Custom email and SMS senders now receive a single props object containing `to`, `code`, `body`, `subject`, and `type`.

### Operational Improvements

- **Project-targeted cache invalidation** - `/clear-cache/:project_id` now clears cache entries by project identifier, making WebUI-driven refreshes more predictable.
- **OpenAuth 0.7.0 update** - The issuer now targets the newer OpenAuth runtime and related provider plumbing.
- **Reduced sensitive request logging** - Authorization header logging has been removed from request logging paths.

## 🔧 Major Improvements

### Global Config Reshape

The configuration surface has been updated to make project-aware customization first-class:

- `openauth.config.ts` now receives both the request context and the current project.
- Authentication hooks can run with request-aware context instead of only environment-level state.
- The global config types were split into a dedicated internal helper surface to support richer runtime behavior.

### More Flexible Authentication Hooks

- **`onSuccessfulAuthentication` hook** - Replaces the older registration-only hook model so custom logic can run for both registration and login.
- **Project-aware access control** - The default config can now make decisions based on the current project, including stricter gating for the public WebUI project.

## 🛡️ Security and Reliability

This release also includes the `v0.3.1` security and bug-fix work, including:

- **TOTP setup expiry fix** - Setup tokens now expire correctly instead of remaining valid indefinitely.
- **Password validation fix** - Uppercase password validation no longer rejects valid strong passwords.
- **Safer cookies** - Session-sensitive cookies now include stronger security attributes.
- **Protected cache clearing** - Cache-clearing routes require authorization instead of accepting anonymous requests.
- **Less sensitive log output** - OTP values, cookie identifiers, and authorization data are no longer logged.
- **More defensive error handling** - Session middleware and signature validation paths now fail more safely.
